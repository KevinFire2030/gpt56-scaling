# 테스트 3A_실습 가이드: Hermes Agent로 지구 vs 달 착지 시뮬레이터 만들기

이 문서는 Dante Labs GPT-5.6 Scaling Lab의 **테스트 3A: 달 착지 비교 시뮬레이터**를 누구나 따라 해 볼 수 있게 정리한 실습 가이드입니다.

목표는 Three.js 기반 단일 HTML 파일 `index.html`을 만들고, 지구와 달에서 같은 힘으로 점프/낙하했을 때 실제 물리 비율이 맞는지 확인하는 것입니다.

중요: 이 실습은 Codex CLI가 아니라 **Hermes Agent 프로파일**로 실행합니다. 각 프로파일은 OpenRouter API 키가 아니라 기본 Codex 인증을 복사해 쓰는 `openai-codex` provider를 사용했습니다.

## 1. 비교 조합

| 조합 | Hermes 프로파일 | provider | model | reasoning |
| --- | --- | --- | --- | --- |

| Sol Ultra | `sol_u` | `openai-codex` | `openai/gpt-5.6-sol` | `ultra` |
| Sol Max | `sol_max` | `openai-codex` | `openai/gpt-5.6-sol` | `max` |
| Sol High | `sol_h` | `openai-codex` | `openai/gpt-5.6-sol` | `high` |

이번 실습에서 `sol_h`, `sol_max`, `sol_u`는 새로 만들었습니다.

```powershell
hermes profile create sol_h --clone-from sol_m --description "Test3A Sol High benchmark agent"
hermes profile create sol_max --clone-from sol_m --description "Test3A Sol Max benchmark agent"
hermes profile create sol_u --clone-from sol_m --description "Test3A Sol Ultra benchmark agent"

hermes -p sol_h config set agent.reasoning_effort high
hermes -p sol_h config set delegation.reasoning_effort high
hermes -p sol_max config set agent.reasoning_effort max
hermes -p sol_max config set delegation.reasoning_effort max
hermes -p sol_u config set agent.reasoning_effort ultra

```

## 5. 공통 프롬프트

프롬프트는 `runs/test3a/_prompt.txt`에 저장한 뒤 Python `subprocess.run([...])`으로 Hermes에 전달했습니다. PowerShell에서 긴 한글 프롬프트와 따옴표를 바로 넘기면 인자 분리가 깨질 수 있기 때문입니다.

핵심 지시:

```text
Three.js로 "지구 vs 달" 착지 비교 시뮬레이터를 만들어 주세요.
외부 파일 없이 브라우저에서 바로 열리는 단일 HTML 파일 하나(index.html)로 완성해야 합니다.

필수:
- 좌우 분할 화면
- 지구 훈련장과 달 표면
- 동일한 우주비행사
- 달 착륙선과 DANTE LABS 로고
- 점프 버튼, 낙하 버튼, 리셋 버튼
- [EARTH]/[MOON] jump_apex_m=...
- [EARTH]/[MOON] fall_time_s=...
- [EARTH]/[MOON] landed
- VERIFICATION.md 기록
```

## 6. 실행 방법

PowerShell에서 직접 `hermes -z $prompt`를 호출하지 않고 Python 래퍼를 사용했습니다.

```powershell
$env:PYTHONUTF8 = "1"
@'
import json, subprocess, time
from pathlib import Path

root = Path.cwd()
p = "terra_max"
run = root / "runs" / "test3a" / p
run.mkdir(parents=True, exist_ok=True)
prompt = (root / "runs" / "test3a" / "_prompt.txt").read_text(encoding="utf-8")

start = time.time()
rc = subprocess.run(
    ["hermes", "-p", p, "--yolo", "-z", prompt, "--usage-file", ".\\usage.json"],
    cwd=run,
).returncode

rec = {
    "profile": p,
    "seconds": round(time.time() - start, 1),
    "html": (run / "index.html").exists(),
    "usage": (run / "usage.json").exists(),
    "returncode": rc,
}

with (root / "runs" / "test3a" / "run_times.jsonl").open("a", encoding="utf-8") as f:
    f.write(json.dumps(rec, ensure_ascii=False) + "\n")
print(json.dumps(rec, ensure_ascii=False))
'@ | python -
```

`p` 값만 `terra_max`, `sol_u`, `sol_max`, `sol_h`, `luna_max`, `sol_m`으로 바꿔 6회 실행했습니다.

## 8. 실제 테스트 과정에서 나온 문제와 해결 방법

| 문제 | 원인 | 해결 방법 |
| --- | --- | --- |
| PowerShell 직접 실행에서 `hermes: invalid choice: 'vs'` 발생 | 프롬프트의 `"지구 vs 달"` 따옴표가 native command 인자 분리를 깨뜨림 | 프롬프트를 `_prompt.txt`에 저장하고 Python `subprocess.run([...])`으로 단일 인자 전달 |
| `sol_h`, `sol_max`, `sol_u` 프로파일이 없음 | 기존 테스트에서는 Sol Medium만 준비되어 있었음 | `sol_m`을 복제하고 `agent.reasoning_effort`, `delegation.reasoning_effort`를 각각 high/max/ultra로 설정 |
| `sol_u`, `sol_h`, `luna_max`가 usage 기록 없이 종료 전 대기 | 산출물 생성 후 자체 QA/마무리 단계에서 Hermes가 오래 대기 | `index.html`과 검증 산출물이 있으면 세션을 중단하고 채점 진행. 토큰/비용은 측정 실패로 표기 |
| 여러 산출물이 “낙하 버튼” 대신 점프만 구현 | 원문은 점프와 같은 높이 낙하를 별도 요구하지만 모델이 점프 체공 시간만 `fall_time_s`로 기록 | 공식 채점에서 `G3_buttons_clicked` 또는 `G2_gravity_checks` 감점. 다음 재현에서는 `DROP TEST` 버튼과 8m 낙하 로그를 명시 강화 필요 |
| `fall_time_s` 정의가 채점기 기대와 어긋남 | 일부 모델은 전체 체공 시간이 아니라 정점 이후 낙하 시간만 출력 | 채점기는 달/지구 낙하 시간비 약 2.46을 기대. 점프 전체 체공 시간 또는 정점 이후 시간 중 무엇을 출력할지 프롬프트에 더 엄밀히 적어야 함 |
| `terra_max`, `sol_max`, `luna_max`, `sol_m`은 채점 중 로그 미확보 | 버튼 텍스트/이벤트 타이밍/로그 출력 위치가 채점기 클릭 루프와 맞지 않음 | `VERIFICATION.md`에는 자체 측정값이 있으나 공식 자동 채점은 실패. 버튼 이름을 `jump`, `drop`, `fall`과 함께 병기하는 것이 좋음 |
| `sol_h`가 실행 폴더에 `node_modules/`와 QA PNG를 생성 | 자체 Playwright QA를 위해 로컬 패키지를 설치하고 스크린샷을 남김 | 채점에는 영향 없음. 실행 산출물은 커밋하지 않음 |
